# FHIR Definitions to Malli ‚Äî Architecture Notes

## Overview

This project generates Clojure malli schemas from FHIR R4B StructureDefinitions. The main entry point is `(-main "R4B")` which imports JSON definition files and writes `.cljc` files containing malli schema ASTs.

## Key File

`src/com/breezeehr/fhir_defintions_to_malli.clj` ‚Äî the sole source file (~1300 lines).

## Import Pipeline

### Data Sources (in processing order)
1. `profiles-types.json` ‚Äî primitive and complex types (Element, BackboneElement, CodeableConcept, Quantity, etc.)
2. `profiles-resources.json` ‚Äî base resources (Patient, Observation, etc.)
3. `extension-definitions.json` ‚Äî 398 FHIR extensions (newly added, not yet fully working)
4. `profiles-others.json` ‚Äî constrained profiles (vitalsigns, bp, bmi, etc.)

### `dependency-order` processes definitions in waves by `baseDefinition` URL, ensuring parents are imported before children.

### `structure-definition->schema` is the core function that processes each StructureDefinition:
- Creates a keyword like `:org.hl7.fhir.StructureDefinition.v4-3-0/Observation`
- For simple extensions: uses `simple-extension->patch`
- For profiles with a base: patches the base schema via `structure-definition->patch`
- For standalone types: builds schema from scratch
- Stores result in `*schema-atom*` with keys `:sch`, `:malli-sch`, `:form`, `:kind`, `:references`, `:direct-references`

## Schema Accumulator

The accumulator `acc` flows through transductions with two keys:
- `:sch` ‚Äî live malli schema object (used for runtime schema operations)
- `:form` ‚Äî vector of Clojure code forms (used to generate the output `.cljc` file)

Additional keys added during slicing:
- `:pending-slicing` ‚Äî map of `{base-field-kw -> {:discriminators, :rules, :base-form, :base-sch, :slices}}`
- `:slicing-info` ‚Äî not currently used (captured via `:pending-slicing` instead)

## Element Processing Pipeline

```
structure-definition->patch
  ‚îî‚îÄ transduction: group elements by first 2 path segments
       ‚îî‚îÄ element-definition->attribute (for each group)
            ‚îî‚îÄ apply-element-patch (for each attribute type)
                 ‚îú‚îÄ slice path (sliceName present)
                 ‚îÇ   ‚îú‚îÄ extension slice ‚Üí promote as sibling key (mu/assoc)
                 ‚îÇ   ‚îî‚îÄ non-extension slice ‚Üí collect into :pending-slicing
                 ‚îî‚îÄ non-slice path
                      ‚îú‚îÄ update-existing-child-schema (field exists in parent)
                      ‚îî‚îÄ create-new-child-schema (new field)
                           ‚îî‚îÄ attr->value-schema-patch
                                ‚îî‚îÄ compute-element-patch (multimethod dispatch on type code)
                                     ‚îú‚îÄ "Element" / "BackboneElement" ‚Üí patch-element (recursive)
                                     ‚îú‚îÄ :default with sub-elements ‚Üí patch-element
                                     ‚îú‚îÄ :default without sub-elements ‚Üí prim-or-ref
                                     ‚îî‚îÄ :string, :boolean, etc. ‚Üí identity
```

### `patch-element` is the recursive workhorse:
- Inner transduction groups sub-elements by `(take (inc (count main-path)) path)`
- Completion function wraps form in `(-> Base/sch (m/schema options) m/deref ~@forms)`
- Handles `add-rt?` for resourceType properties on sub-elements
- Flushes pending slicing in completion

## Slicing Implementation

### Two categories:
1. **Extension slices** (`extension:race`, `modifierExtension:...`) ‚Äî promoted as sibling keys on the parent map. Existing behavior, unchanged.
2. **Non-extension slices** (`coding:BMICode`, `component:SystolicBP`) ‚Äî collected during transduction, flushed as `[:sequential [:multi {:dispatch ...} ...entries...]]` in completion function.

### Dispatch function generation:
- FHIR discriminator paths like `code.coding.code` converted to `(get-in m [:code :coding 0 :code])` 
- Sequential fields (`:coding`) get `0` index inserted automatically by walking the schema
- Single discriminator ‚Üí scalar dispatch value; multiple ‚Üí vector
- `rules: "open"` ‚Üí `:default` entry present; `"closed"` ‚Üí no default

### Key challenge: sch context in :multi entries
Inside `(mu/update :component (fn [sch] [:sequential [:multi ...]]))`, the `sch` parameter is the original sequential. Each `:multi` entry needs the **inner** element, not the sequential. Solution:
```clojure
(-> (mu/get sch 0) (m/schema options) m/deref ...)
```
This extracts the inner element, compiles it with the registry, and derefs any `:ref` wrappers.

### Form fixing in `flush-pending-slicing`:
Slice forms generated by `attr->value-schema-patch` may contain:
- `(-> sch ...)` ‚Äî replace `sch` with `(-> (mu/get sch 0) (m/schema options) m/deref)`
- `(mu/update 0 (fn [x] body))` ‚Äî drop entirely (base type replacement, already handled by `mu/get sch 0`)
- `(mu/update sch 0 (fn [x] body))` ‚Äî same, drop it

## Ref Resolution

Malli schemas returned by `mu/get` are schema objects, not vectors. The ref resolution in `patch-element` handles both:
- Vector refs: `[:ref :org.hl7.fhir.../CodeableConcept]` ‚Äî check `(vector? old-sch)`
- Schema object refs: malli schema with `(m/type old-sch) = :ref` ‚Äî check via `m/type`

After resolution, `m/deref` is needed before `mu/update` operations on map keys. Without deref, `mu/update :coding` on a `:ref` schema throws `:malli.core/index-out-of-bounds`.

## Form Generation Patterns

Generated `.cljc` files contain `(reify m/IntoSchema (-into-schema [_ _ _ options] ...))` which builds the schema at compile time. The form uses:
- `(-> Base/sch (m/schema options) m/deref ...)` ‚Äî start from base, compile, deref
- `(mu/assoc :key schema)` ‚Äî add new key
- `(mu/update :key (fn [sch] ...))` ‚Äî modify existing key
- `(mu/update-properties merge {...})` ‚Äî update schema properties
- `(mu/update-entry-properties :key merge {...})` ‚Äî update entry metadata

## Known Issues

1. **Extension import**: `extension-definitions.json` is processed but extensions don't appear in schema-atom. Needs investigation ‚Äî likely `simple-extension?` classification or `basekw` resolution issue.
2. **lipidprofile**: "Key must be integer" error ‚Äî edge case in sequential handling
3. **shareablecodesystem**: `:malli.core/invalid-ref` for contentReference `#CodeSystem.concept`
4. **`clj-paren-repair`**: Corrupts `cond->` blocks by merging the test's closing `)` with the expression. Never use it on this file. Use `clj-kondo` for validation instead.
